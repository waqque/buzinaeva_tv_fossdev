# Автоматизация разработки на Python с Makefile

## О проекте

Проект демонстрирует подход к автоматизации рутинных задач разработки на Python с использованием Makefile. Вместо выполнения команд вручную, разработчик получает единую точку входа для управления окружением, зависимостями, проверкой качества кода и запуском приложения.

Цель — показать, как Makefile может служить воспроизводимым интерфейсом между разработчиком и проектом, устраняя необходимость запоминать длинные команды и последовательности действий.

## Структура репозитория

```
.
├── Makefile                 # Центральный файл автоматизации
├── requirements.txt         # Зависимости проекта
├── .gitignore              # Исключения для Git
├── README.md               # Документация
├── src/                    # Исходный код
│   ├── app.py             # Основное приложение
│   ├── calc.py            # Модуль с вычислениями
│   ├── example.py         # Пример с форматированием
│   └── service.py         # Сервисный модуль
└── scripts/                # Вспомогательные скрипты
    └── check_requirements.py # Проверка зависимостей
```

## До автоматизации

Без Makefile разработчик вынужден выполнять последовательность команд, которая может отличаться в зависимости от окружения и опыта.

**Создание окружения:**
```bash
python -m venv .venv
```

**Активация окружения:**
```bash
# Windows
.venv\Scripts\activate
# Linux/macOS
source .venv/bin/activate
```

**Установка зависимостей:**
```bash
pip install -r requirements.txt
```

**Запуск приложения:**
```bash
python src/app.py
```

**Проверка типов:**
```bash
pip install mypy
mypy src/
```

**Форматирование кода:**
```bash
pip install black
black src/
```

**Проверка стиля:**
```bash
pip install flake8
flake8 src/
```

**Проверка зависимостей:**
```bash
python scripts/check_requirements.py
```

**Проблемы ручного подхода:**
- Необходимость запоминать команды
- Разные команды для разных ОС
- Риск использовать глобальное окружение вместо проектного
- Отсутствие единого стандарта в команде
- Сложность воспроизведения на другом компьютере

## После автоматизации

Все действия объединены в Makefile. Достаточно знать несколько команд, которые работают одинаково на всех платформах.

### Команды Makefile

| Команда | Что делает | Результат |
|---------|------------|-----------|
| `make install` | Создаёт .venv, устанавливает зависимости | Готовое окружение |
| `make run` | Запускает приложение | Вывод программы |
| `make check-deps` | Проверяет импорты против requirements.txt | Отчёт о недостающих зависимостях |
| `make typecheck` | Запускает mypy для проверки типов | Отчёт об ошибках типов |
| `make format` | Форматирует код через black | Исправленный код |
| `make lint` | Проверяет стиль через flake8 | Отчёт о нарушениях |
| `make check` | Запускает typecheck, check-deps, lint | Итоговая проверка |
| `make clean` | Удаляет .venv и временные файлы | Чистый репозиторий |

### Использование

```bash
# Первый запуск
make install

# Запуск приложения
make run

# Проверка качества кода
make check

# Если нужно отформатировать код
make format

# Полная очистка
make clean
```

## Реализация

### Работа с окружением (stateless)

В Makefile используется явный путь к интерпретатору:

```makefile
VENV := .venv
PYTHON := $(VENV)/Scripts/python
PIP := $(VENV)/Scripts/pip
```

Это гарантирует, что команды выполняются именно в проектном окружении, а не в глобальном. Активация через `source` не используется, так как каждая строка Makefile выполняется в отдельном shell и состояние не сохраняется.

### Проверка зависимостей

Скрипт `scripts/check_requirements.py` анализирует все импорты в `src/` и сравнивает с `requirements.txt`. Если обнаружены импорты, не указанные в зависимостях, команда завершается с ошибкой.

Пример:
```python
import requests    # указано в requirements.txt
import numpy       # не указано -> ошибка
```

### Проверка типов

Используется mypy для статического анализа. В исходном коде присутствовала ошибка:

```python
result: int = add(2, "3")  # строка вместо int
```

После исправления:
```python
result: int = add(2, 3)
```

### Проверка стилей

- `make format` — автоматически исправляет код через black
- `make lint` — только проверяет через flake8, не изменяя файлы

### Композиция задач

Target `check` объединяет несколько проверок:

```makefile
check: typecheck check-deps lint
```

Выполнение одной команды запускает последовательно проверку типов, зависимостей и стиля. При ошибке любого этапа процесс останавливается.

## Результаты

### После `make install`

- Создана папка `.venv/`
- Установлены все зависимости из requirements.txt

### После `make check`

- Проверка типов: ошибок нет
- Проверка зависимостей: все импорты объявлены
- Проверка стиля: код соответствует стандартам

### После `make clean`

- Папка `.venv/` удалена
- Все `__pycache__/` и `*.pyc` удалены
- Исходный код остаётся неизменным

### Проверка чистоты репозитория

```bash
make clean
git status --porcelain
```

Команда не показывает изменений — все артефакты сборки находятся в `.gitignore`.

## Требования к окружению

- Python 3.8+
- Make (для Windows: через Git Bash или установленный отдельно)

## Заключение

Makefile превращает набор разрозненных команд в единый воспроизводимый интерфейс. Это снижает порог входа для новых разработчиков, исключает ошибки при ручном вводе и обеспечивает единообразие процессов в команде.

Проект демонстрирует ключевые практики:
- Изоляция окружения через venv
- Автоматическая проверка зависимостей
- Статический анализ типов
- Автоматическое форматирование кода
- Композиция проверок через зависимости target'ов
